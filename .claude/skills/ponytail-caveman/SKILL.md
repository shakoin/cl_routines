# Ponytail + Caveman

Use this skill when producing recurring reports, digests, scouts, or deep sweeps.

## Purpose

Reduce unnecessary work and unnecessary words.

- Ponytail = do not build, search, or report what does not need to exist.
- Caveman = say the same technical meaning with fewer words.

The goal is not a novelty voice. The goal is low noise, low token use, and high usefulness.

## Ponytail scope ladder

Before researching or writing an item, ask:

1. Does this matter to the user today or this week?
2. Does it affect action, risk, decision, or watchlist?
3. Is there a primary source?
4. Is one source enough?
5. Can this be one line?
6. Can this be omitted safely?

If the item does not affect action, risk, decision, or watchlist, skip it.

## Ponytail section ladder

Before writing a section, ask:

1. Did this section produce an S3/S2/S1 item?
2. Did a source fail?
3. Is there an unresolved watch item?
4. Is this required for the report structure?

If all are no, collapse to one quiet line or omit details.

## Caveman writing rules

Use compact operational wording.

Preferred item format:

```text
[severity evidence] Subject — Change. Impact. Next step. Source.
```

Examples:

```text
[S3 primary] Example service — Active exploit reported. Version unknown. Next: verify exposure.
[S2 secondary] Example hardware — Price below threshold. Purchase decision affected. Next: compare options.
[S1 community] Example outage — User reports only. Not confirmed. Next: watch; no action.
```

Avoid:

- filler intros;
- repeated context;
- long paragraphs for quiet items;
- generic newsletter language;
- forced importance.

## Quiet-day rule

If no action or decision trigger exists, say so plainly.

```text
No action needed today.
```
